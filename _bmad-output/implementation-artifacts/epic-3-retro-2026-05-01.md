## What Went Well

- The decision to implement Story 3.7 (localhost exemption) before 3.1 eliminated a real Docker health-check failure mode before it could affect the epic.
- Rate-limit middleware (`rate-limit.ts`) was already well-structured with isolated, named limiters, which made testing, exemption logic, and documentation straightforward.
- The `authenticatedKeyGenerator` compound key design (`scope|org|username|id`) prevented cross-tenant counter sharing without any story-time change; we documented and tested a solid pre-existing contract.
- Story 3.8 (documentation-only) kept scope strictly bounded: scan README, insert once, verify build. No scope creep.
- `NODE_ENV=test` skip behavior was consistently respected across all rate-limit stories, keeping test suites deterministic.
- Epic 2 Redis/Testcontainers infrastructure was reused effectively for integration tests in SSE and rate-limit stories.

## Challenges

- The SSE concurrent load test (Story 3.4) required careful tuning to avoid false positives from test environment resource limits rather than application limits. Fan-out assertions at 50 clients are sensitive to CI timing.
- Story 3.6 (window reset validation) exposed a UI timing dependency: the countdown timer in `TenantProvider` required real clock progression, making the test slower than typical integration tests. `skipSuccessfulRequests` semantics needed explicit documentation to avoid future confusion.
- The boundary between `generalLimiter` (all `/api/*` routes) and `protectedLimiter` (specific authenticated route groups) was not obvious from route files alone; understanding the layered application required reading both `app.ts` and individual route files.
- BMAD artifact sync after PR merge remains a manual step (sprint-status, story status, epic status), requiring a dedicated follow-up each time.

## Key Discoveries

### Architecture Discovery

The rate-limit middleware is layered: `generalLimiter` applies globally to all `/api/*` routes in `app.ts`, while `protectedLimiter`, `scraperLimiter`, etc. apply on top of that in individual route files. A protected endpoint is effectively subject to two limiters. The `generalLimiter` limit (100 req / 15 min) is intentionally much looser than `protectedLimiter` (60 req / 15 min) so the global layer only catches genuinely abusive traffic.

### Testing Discovery

`isTrustedLocalHealthProbe` requires all three conditions simultaneously (req.ip, socket remoteAddress, and X-Forwarded-For chain). Testing the exemption requires constructing a request that satisfies all three, which is more precise than simply mocking `req.ip`. This nuance should be preserved in future tests.

### Process Discovery

Documentation-only stories benefit from an explicit "scan before writing" subtask. The instruction to grep README for existing sections before inserting content prevented duplication and identified the correct anchor point (`## Runtime Notes` → `## Documentation`).

## Lessons Learned

1. Limiter layering should be called out explicitly in story Dev Notes for any story that touches rate-limit behavior, so implementors understand which limiter is the binding constraint.
2. Tests that depend on real clock progression (e.g., window reset) should be isolated in a serial test project and flagged for expected slowness rather than being treated as flaky.
3. The "scan first, insert once" discipline for documentation stories prevents duplication and keeps README coherent; make it a subtask template.
4. BMAD artifact closure (story → review → done, sprint-status, epic status) should be part of the post-merge checklist, not an afterthought.

## Repeated Patterns Across Stories

- Every SSE and rate-limit story reused the same Redis/Testcontainer infrastructure without modification — Epic 0/2 foundations continue to pay dividends.
- Rate-limit tests consistently required `NODE_ENV=development` or `production` mode; the skip behavior in `test` mode was surfaced multiple times and is now documented.
- Review findings consistently focused on boundary completeness (exemption edge cases, X-Forwarded-For chain coverage, per-tenant isolation) rather than correctness of the happy path.
- Cross-tenant counter isolation (JWT compound key) was mentioned in every rate-limit story as a critical invariant.

## Technical Debt and Follow-Up Risks

- `generalLimiter` is also applied to the SPA static fallback handler in `app.ts`; this is intentional but undocumented outside the code. A future story could add a brief note to the README "Rate Limiting" section clarifying this.
- The Admin API (`rate_limit_configs`) takes effect only on server restart; this restart requirement is noted in the README but not enforced anywhere. A future story could add a server log warning or admin UI indicator.
- CI timing sensitivity in Story 3.4's 50-client SSE fan-out test may surface as flakiness under CI load spikes. Monitor before assuming stable.
